## 项目名称
多 Agent 短剧剧本生成平台

## 项目简介
这是一个面向短剧内容生产场景的大模型应用项目，目标不是简单做一个“提示词调用器”，而是把剧本生成拆解成多个职责明确的环节，形成“策划大纲 → 剧本生成 → 审校修订 → 导出交付”的完整生产链路。系统支持用户在网页端输入主题、关键词、受众、风格、集数、单集时长等信息，后端使用 `Director + Planner + Writer + Reviewer` 的多 Agent 协作方式生成结果，并输出可下载的 Word / PDF 文件。

## 为什么这个项目有价值，而不是只做简单 Prompt 工程
### 1. 简单 Prompt 可以生成文本，但很难稳定生成“可交付结果”
单条 Prompt 的确可以直接让模型写剧本，但在真实业务里会遇到几个问题：

- **[问题一]** 输出结构不稳定  
  同样的输入，有时会缺场景号、有时没有对白字段、有时结尾直接截断。

- **[问题二]** 长文本更容易失控  
  多集剧本场景数更多，单轮生成容易出现漏场景、重复结构、后半段质量下降。

- **[问题三]** 缺少质量闭环  
  只靠一次生成，无法系统判断“是否真的完整、是否满足场景数、是否符合风格、是否可导出交付”。

- **[问题四]** 工程不可控  
  简单 Prompt 工程更像一次性试验，不像一个可监控、可复用、可演进的产品系统。

### 2. 这个项目真正解决的是“大模型应用工程化”问题
这个项目的价值不只是“让模型写一段文本”，而是把模型能力做成一个有状态、有路由、有校验、有失败处理的应用系统：

- **[工程化编排]** 把创作过程拆成 Planner、Writer、Reviewer 三类角色，减少单轮 Prompt 承担过多任务
- **[可控输出]** 通过场景数校验、完整性判断、审校反馈、定向补尾，把结果从“能生成”提升到“更稳定可用”
- **[可观测性]** 前端可看到 Director 调度日志，后端可拿到 `writer_mode`、`review_mode`、`script_status`
- **[可交付]** 最终不仅输出文本，还能导出 DOCX / PDF，进入实际内容生产链路

### 3. 面试里可以这样回答
如果面试官问“为什么不直接用 Prompt”，可以回答：

> 单 Prompt 更适合验证想法，但这个项目面向的是可交付的长文本内容生产。短剧剧本对结构完整性、场景数、风格一致性、结尾闭环要求都更高，所以我把问题拆成规划、生成、审校、修订几个环节，用多 Agent + 状态流把生成过程工程化，核心价值在于提升可控性、稳定性和可交付性，而不是只追求一次生成出字。

## 这个项目和 LangChain、LangGraph 的关系
### 1. 关系是什么
这个项目的核心编排框架是 `LangGraph`。  
在代码里，`DirectorAgent` 通过 `StateGraph` 定义了完整状态图，节点包括：

- `planner`
- `writer`
- `reviewer`
- `rewrite`
- `finalize`

状态从 `START` 开始，经由 Planner → Writer → Reviewer 流转，再根据审校结果决定：

- 直接结束
- 进入 rewrite
- 或在达到修订上限后 finalize

也就是说，这个项目不是简单的顺序调用，而是一个**有状态、有条件路由的图式工作流**。

### 2. 为什么要用 LangGraph
因为这个项目天然不是线性的：

- 审校通过就结束
- 审校不通过可能进入重写
- 某些问题只需要补尾，不需要整稿重写
- 达到修订次数上限后，要输出当前最优稿，而不是无限重试

这类问题如果只用普通函数串起来，也能写，但会越来越像手写状态机，后期不容易维护。  
`LangGraph` 的优势是：

- **[状态管理清晰]** 把 `request`、`outline`、`draft`、`review`、`revision_count`、`latest_feedback` 放进统一状态
- **[条件路由清晰]** 审校后去哪条边，由 `_review_route` 明确控制
- **[可扩展性更强]** 后续要加“事实核查 Agent”“素材检索 Agent”“角色设定 Agent”会更自然

### 3. LangChain / LangGraph 在这个项目里用了哪些东西
从当前实现看：

- **[LangGraph]** 用了 `StateGraph`、`START`、`END`、条件边、节点编排
- **[LangChain 生态关系]** 项目更偏 LangGraph 的工作流编排思想，而不是大量使用 LangChain 的链式组件
- **[LLM 调用层]** 模型调用本身是项目自定义的 `LLMService`，通过 OpenAI 兼容 SDK 接入模型，不是直接依赖 LangChain 的 `LLMChain`

所以面试里更准确的说法是：

> 这个项目主要使用的是 LangGraph 做多 Agent 工作流编排；模型调用、JSON 解析、超时重试、fallback、导出服务等工程能力是我自己封装的。

### 4. 为什么它可以称为 Agent
因为这里不是单一模型回答一次问题，而是每个角色承担不同职责，并基于上下文状态做决策：

- **Planner Agent** 负责把用户需求转成结构化大纲
- **Writer Agent** 负责根据大纲生成正文，必要时做定向补尾
- **Reviewer Agent** 负责审校逻辑、结构、风格和完整性
- **Director Agent** 负责调度、路由、状态控制和修订次数管理

这里的 Agent，不一定要求每个角色都具备“自主工具调用”能力；更核心的是：

- 有独立职责
- 有独立输入输出
- 会基于状态协作
- 整体能完成单个模型难稳定完成的复杂任务

### 5. Agent 解决了什么问题，好在哪
- **[降低单次 Prompt 复杂度]** 把“大纲设计 + 正文生成 + 质量审查 + 修订策略”拆开
- **[更容易控质量]** Reviewer 能发现截断、缺场景、风格漂移等问题
- **[更适合长链路任务]** 多集剧本天然比普通问答更适合 Agent 分工
- **[更利于演进]** 后续加检索、知识库、角色设定、审片规则都更容易接入

面试里可以概括成一句：

> Agent 的价值不在于“看起来高级”，而在于把复杂任务拆成可控子任务，提升长链路大模型应用的稳定性和可维护性。

## PDF / Word 导出是怎么做的
### 1. 导出是不是 LLM 驱动的 Agent
不是。  
这个项目里的 PDF / Word 导出是**确定性工程模块**，不是由 LLM 再生成一次文档。

当前实现里，`app.py` 在剧本通过完整性校验后，直接调用：

- `export_service.export_docx(...)`
- `export_service.export_pdf(...)`

真正负责导出的代码在 `services/export_service.py`，使用的是：

- `python-docx` 生成 Word
- `reportlab` 生成 PDF

所以这是**程序编排导出**，不是“让大模型帮你生成一个 PDF”。

### 2. 具体是怎么做的
- **[DOCX]**  
  把标题作为 heading，把剧本内容按行拆分成段落，统一设置中文字体和字号，再保存为 `.docx`

- **[PDF]**  
  先注册中文字体，再按页宽逐字换行，控制页边距、行高和分页，最后输出 `.pdf`

### 3. 为什么不交给 LLM 做导出
因为导出本质上不是语义生成问题，而是**格式化渲染问题**。  
如果把导出也交给 LLM，会有几个问题：

- 不稳定，格式不一致
- 成本更高，没必要多一次模型调用
- 不利于精确控制字体、分页、换行、文件名
- 出现乱码、截断、分页问题时也很难定位

所以更合理的做法是：

- **LLM 负责生成内容**
- **工程模块负责把内容转成文档**

这也是大模型应用开发里常见的边界划分方式。

### 4. 跟直接输出文本有什么区别
直接输出文本只解决“看见内容”的问题；导出 Word / PDF 解决的是“交付和流转”的问题。

- **[直接文本输出]** 适合预览、调试、快速查看
- **[Word 输出]** 适合继续编辑、批注、协作改稿
- **[PDF 输出]** 适合固定版式、归档、发送给外部团队

所以简历和面试里可以强调：

> 我做的不只是一个文本生成 Demo，而是把生成结果接到了实际交付链路里，让内容能被编辑、归档和分发。

## 结合当前项目实现，可重点强调的工程亮点
### 1. 多 Agent 工作流编排
- 使用 `LangGraph StateGraph` 实现 `Planner → Writer → Reviewer → Rewrite / Finalize` 的状态图
- Director 统一维护状态、日志、审校路由和修订次数

### 2. 长文本完整性治理
- 对剧本进行场景数校验、结尾完整性校验、截断检测
- 不完整结果不会误标为成功，而是保留部分结果并返回失败态

### 3. 多集剧本优化
- 支持按 `episodes × episode_duration` 计算目标场景数
- 修复多集请求被错误压缩为单集的问题
- 优化多集补尾逻辑，优先定向补尾而不是整稿重写

### 4. 生成链路性能优化
- 规划阶段从“远程 JSON 修复重试”改为“单次生成 + 本地解析 + 失败快速 fallback”
- 按场景数动态控制 Writer token 策略，兼顾单集速度与多集完整度

### 5. 前后端协同
- 前端轮询任务状态，展示 Director 调度日志
- 前端结构化展示剧本内容，兼容多种场景标题格式
- 后端返回 `review`、`generation`、`script_status` 等元信息，增强可观测性

## 面试问题参考回答
### 1. 这个项目好在什么地方？为什么不直接做 Prompt 工程？
这个项目的价值不在“让模型能写剧本”，而在“让剧本生成变成可控、可监控、可交付的系统”。  
单 Prompt 可以快速做 Demo，但在多集、长文本、强结构要求的场景里，容易出现截断、漏场景、风格漂移和失败不可见的问题。这个项目通过多 Agent 分工、LangGraph 状态流、审校修订、完整性校验和导出交付，把一次模型调用升级成了完整的大模型应用链路。

### 2. 这个项目和 LangChain、LangGraph 有什么关系？为什么要用它们？
这个项目主要使用的是 `LangGraph`，用它来编排多 Agent 工作流和条件路由。因为剧本生成不是线性的：审校通过就结束，不通过可能重写，某些情况只需要补尾，达到上限后要输出当前最优稿。用 LangGraph 可以把这些状态和边表达得很清楚。模型调用层则是自定义的 `LLMService`，用于控制 API、超时、重试和 JSON 解析。

### 3. 为什么它是 Agent？解决了什么问题？Agent 好在哪？
因为这里每个角色都有明确职责和输入输出：Planner 负责策划，Writer 负责生成，Reviewer 负责审校，Director 负责调度。Agent 的好处是把复杂任务拆解成几个更可控的子任务，尤其适合长文本、强结构、需要自我修正的场景。它解决的是单轮 Prompt 难以同时兼顾“结构、质量、完整性和稳定性”的问题。

### 4. PDF、Word 是怎么做的？是 LLM 驱动还是程序编排？
是程序编排，不是 LLM 驱动。项目在剧本通过完整性校验后，调用 `ExportService` 把文本导出为 Word 和 PDF。Word 通过 `python-docx` 构造标题和段落，PDF 通过 `reportlab` 处理中文字体、换行和分页。这样做比让 LLM 参与导出更稳定、成本更低、格式更可控。

### 5. 导出 PDF / Word 和直接输出文本有什么区别？
直接输出文本更适合预览；导出 Word / PDF 代表项目已经具备交付能力。Word 适合编辑协作，PDF 适合归档和对外发送。它体现的是一个大模型应用从“能生成”到“能落地”的差异。

## 项目实现过程
### 1. 需求拆解阶段
项目一开始并不是直接进入编码，而是先把“短剧剧本生成”这个问题拆解成几个更明确的子问题：

- **[输入层]** 用户到底需要提供哪些信息，才能让模型更稳定地生成剧本
- **[生成层]** 单轮生成是否足够，还是需要分步骤处理
- **[质量层]** 怎么判断生成内容是否完整、是否可用
- **[交付层]** 结果是否只是展示文本，还是要支持文档导出

基于这个拆解，我最终把系统分成：

- 用户请求构建
- Planner 大纲生成
- Writer 正文生成
- Reviewer 审校修订
- Director 编排调度
- ExportService 文档导出

### 2. 最初实现阶段
最早的核心目标是先打通从前端输入到后端出稿的基本链路，因此优先实现了：

- Flask 接口与异步任务机制
- 用户请求结构化封装
- LLMService 模型调用封装
- Planner / Writer / Reviewer 三类 Agent
- Director 统一调度
- 前端轮询任务状态与展示结果

这一阶段的重点是把“能跑通”建立起来，让系统具备最基础的可用性。

### 3. 工程化增强阶段
在基本链路打通后，项目开始暴露出典型的大模型应用问题：

- 多集请求容易被压缩成单集
- 剧本偶尔会在尾部截断
- 审校 fallback 会误把问题稿件标成通过
- 前端展示结构化剧本时，对场景标题格式依赖太强
- 长文本请求耗时偏高

针对这些问题，后续重点做了几轮工程化增强：

- **[完整性治理]** 增加场景数校验、结尾完整性校验、截断检测
- **[失败态治理]** 不完整剧本直接标记为失败，不再误报成功
- **[多集优化]** 让场景目标数和 `episodes × episode_duration` 对齐
- **[补尾优化]** 修订时优先定向补尾，而不是整稿重写
- **[Planner 提速]** 去掉多余的远程 JSON 修复调用，改成本地解析 + 快速 fallback
- **[前端兼容]** 放宽场景标题解析规则，提升结构化展示稳定性
- **[导出增强]** 处理中文字体、换行、分页等文档导出问题

### 4. 验证与迭代阶段
实现后并不是只看“能不能出文本”，而是通过真实请求去验证：

- 单集是否能完整生成
- 多集是否会缺场景
- 补尾是否会重复结构
- 不完整结果是否被正确标为失败
- 前端是否能完整显示结果
- 导出的 PDF / Word 是否可读、可交付

这一步体现的是大模型应用开发和普通脚本调用的差异：不是只验证模型有没有输出，而是验证整条应用链路是否可靠。

## 实现思路
### 1. 用“任务拆解”替代“超长 Prompt”
实现思路的核心是：不要把所有要求都塞进一个大 Prompt，而是把复杂任务拆解成多个角色：

- Planner 先做结构化规划
- Writer 根据规划生成正文
- Reviewer 负责审校与问题发现
- Director 负责决定是结束、重写还是补尾

这样做的原因是，短剧剧本属于长文本、强结构、强约束场景，单轮 Prompt 更容易失控。

### 2. 用状态图替代硬编码顺序调用
项目没有采用简单的线性函数链，而是用 LangGraph 建模状态图。  
原因在于剧本生成不是固定顺序流程，而是一个带条件分支的工作流：

- 审校通过就结束
- 审校不通过则重写
- 尾部不完整优先补尾
- 达到修订次数上限后输出当前最优稿

因此，更适合用状态图管理而不是写成大量 `if/else` 嵌套逻辑。

### 3. 用“规则校验 + LLM 修正”组合提升稳定性
这个项目没有完全依赖模型自我纠错，而是采用：

- **[规则校验]** 场景数、标题格式、结尾完整性、是否截断
- **[LLM 修正]** 根据审校意见执行整稿重写或定向补尾

这种组合式思路更适合工程场景，因为：

- 规则校验稳定、便宜、可解释
- LLM 修正灵活、适合处理语义问题

### 4. 用“内容生成”和“文档导出”分层设计
实现思路上，系统明确区分了两层能力：

- **[生成层]** LLM 负责生成大纲、正文和审校结果
- **[交付层]** 程序模块负责导出 Word / PDF

这样设计的优点是职责边界清晰，导出问题不会污染生成逻辑，生成质量问题也不会和文件格式问题混在一起。

### 5. 用“可观测性”支持调试和迭代
为了便于排查问题，项目在实现时保留了较强的观测信息：

- Director 调度日志
- `writer_mode`
- `review_mode`
- `script_status`
- 前端任务状态轮询

这让系统不仅能“运行”，还能“解释为什么成功或失败”。

## 涉及技术栈
### 1. 后端与服务框架
- **Python**

## 从用户自定义输入到剧本生成再到视频生成的完整链路
### 1. 用户输入层：把模糊创意转成结构化创作请求
这个项目的入口不是固定题材模板，而是用户自定义输入。用户可以在前端输入：

- **主题**
- **关键词**
- **目标受众**
- **风格**
- **集数**
- **单集时长**
- **补充要求**

系统会先把这些输入整理为统一的 `UserRequest`，作为后续多 Agent 工作流的标准输入。  
这一步的核心价值是：把“我想要一个什么样的短剧”从自然语言需求，转成可以被模型和工作流稳定消费的结构化请求。

### 2. 规划层：先产出大纲，而不是直接写全文
系统不会直接把用户输入丢给 Writer 生成整稿，而是先进入 Planner 阶段，先生成：

- **标题**
- **开场钩子**
- **核心冲突**
- **三幕结构**
- **反转点**
- **结尾钩子**

这样做的原因是，短剧剧本属于长文本、强结构任务，如果没有中间规划层，正文生成更容易出现结构漂移、节奏失控和结尾崩坏。

### 3. 正文生成层：按约束产出可交付剧本
在拿到结构化大纲后，Writer Agent 会根据：

- 用户主题与关键词
- 目标受众与风格
- 集数与单集时长
- Planner 生成的大纲
- 修订反馈

输出完整剧本。  
当前 Writer 已经进一步升级为更适合后续视频化处理的模板，能稳定产出：

- **场景号**
- **内外景**
- **时间**
- **地点**
- **角色造型&情绪**
- **动作描述**
- **对白**
- **分镜提示**

其中 `分镜提示` 不只是简单写“镜头推进”，而是尽量包含：

- **景别**
- **运镜**
- **转场**
- **焦点人物**
- **表情**
- **微动作**

这样做的意义在于，剧本不再只是给人阅读，也能作为后续分镜解析和视频生成的中间表示。

### 4. 审校与修订层：把“能生成”升级成“更稳定可用”
正文生成后不会直接交付，而是进入 Reviewer 和 Director 控制的质量闭环。  
系统会结合规则判断和审校反馈去检查：

- 场景数是否达标
- 是否存在尾部截断
- 最后一场是否完整
- 风格是否偏移
- 是否满足用户主题和关键词要求

如果发现问题，不会直接整稿推翻，而是优先采用：

- **定向补尾**
- **局部修补**
- **有限次重写**

这一步体现的是大模型应用开发里的“生成治理”能力，而不是只让模型一次性吐结果。

### 5. 视频结构化层：把剧本文本转成镜头级数据
在剧本通过基础质量校验后，系统会进一步进入视频化处理流程。  
`ScriptParser` 会把剧本文本解析成：

- `Scene`
- `Shot`
- `CharacterProfile`
- `DialogueLine`

并补充适合视频生成的镜头级元数据，例如：

- **角色外貌**
- **服装**
- **气质**
- **表情**
- **肢体动作**
- **场景环境**
- **景别**
- **运镜**
- **转场**
- **旁白文本**
- **TTS 文本**

这样，系统中间不再是“只有一大段剧本文本”，而是变成了可驱动后续图像、语音、视频模块的结构化 storyboard 数据。

### 6. 多模态生成层：从镜头脚本到视频资产
在视频生成阶段，系统当前已经实现了从剧本直接驱动视频资产生成的基础链路：

- **文生图 / 关键帧生成**
- **角色对白与旁白 TTS**
- **镜头级视频片段渲染**
- **字幕时间轴生成**
- **最终成片拼接**

整体路径可以概括为：

`用户输入 → 多 Agent 剧本生成 → 剧本结构化解析 → 镜头资产生成 → 时间轴构建 → FFmpeg 成片渲染`

这说明项目已经从传统“文本生成系统”扩展到了“文本驱动多模态内容生产系统”的方向。

## 视频化实现思路、技术路线与当前效果
### 1. 设计思路：用“结构化中间表示”连接大模型与多模态渲染
我在视频化扩展里坚持的核心原则是：

- **LLM / Agent 负责生成结构和语义**
- **工程模块负责资产生产与渲染交付**

也就是说，不会把整份剧本一次性扔给某个视频模型赌结果，而是先把剧本变成可解析的分镜级中间表示，再驱动多个子模块协同工作。  
这个设计更适合工程落地，因为它：

- 更容易定位问题
- 更容易局部重试
- 更容易控制角色一致性和时间轴
- 更方便替换底层图像、TTS、视频 provider

### 2. 技术路线：从文本生成系统扩展为多模态流水线
当前技术路线大致分成五层：

#### 第一层：用户需求建模
- 前端收集主题、关键词、风格、时长等信息
- 后端将其封装为结构化请求对象

#### 第二层：多 Agent 剧本生成
- `DirectorAgent` 负责状态流编排
- `PlannerAgent` 负责生成故事骨架
- `WriterAgent` 负责生成视频友好的场景剧本
- `ReviewerAgent` 负责审校与修订反馈

#### 第三层：剧本结构化解析
- `ScriptParser` 将剧本文本转成 `Scene / Shot / CharacterProfile`
- 补齐人物造型、镜头语言、旁白策略、TTS 文本、图像提示词和视频提示词

#### 第四层：镜头资产生产
- `ImageProvider` 生成镜头关键帧
- `TTSProvider` 根据人物角色和性别分配音色，生成对白或必要旁白
- `VideoProvider` 基于关键帧、音频和运镜规则渲染单镜头视频

#### 第五层：时间轴与最终渲染
- `VideoPipelineService` 负责组织 storyboard、assets、timeline、render plan
- 生成字幕文件、concat 文件和最终成片
- 用 `FFmpeg` 完成镜头拼接、转场处理和字幕烧录

### 3. 关键实现点
#### 3.1 剧本模板升级为“视频友好型”
不是只让 Writer 写可读剧本，而是要求它直接产出更适合解析的场景模板。  
这样后续不需要再做大量不稳定的信息抽取，解析器可以更稳地拿到：

- 场景信息
- 人物外观
- 情绪和动作
- 镜头语言

#### 3.2 旁白策略从“全覆盖”改成“按镜头类型选择”
项目里已经对 TTS 策略做了优化，不再简单地把所有无对白镜头都配成旁白，而是区分：

- **对白镜头**
- **纯视觉镜头**
- **需要叙事补充的旁白镜头**

这样可以避免成片里出现“每个沉默镜头都被强行念一遍画面描述”的问题，整体观感会更接近真实短视频叙事节奏。

#### 3.3 音画同步按真实时长对齐
视频化里一个非常核心的问题是音画同步。  
当前实现已经把时间轴逻辑统一到真实音频时长上：

- 单镜头先生成真实音频
- 再按真实音频时长裁切镜头视频
- 时间线按实际片段时长累计
- 字幕时间轴与最终成片时长对齐

这一步解决的不是“能不能生成一个视频”，而是“生成后是否能正常播放、字幕和镜头是否基本同步”。

#### 3.4 Provider 抽象支持后续替换与扩展
视频链路没有把外部能力硬编码死，而是抽象成 provider：

- `ImageProvider`
- `TTSProvider`
- `VideoProvider`

这样可以很自然地从占位实现切换到真实 API，也可以在后续接入新的图像模型、TTS 服务、视频模型或本地渲染方案。

### 4. 当前实现效果
从当前项目状态看，已经完成了从“用户输入生成剧本”到“基于剧本直接生成视频”的 MVP 链路，具备以下效果：

- **支持用户自定义输入生成剧本**
- **支持多 Agent 协作完成规划、生成、审校与修订**
- **支持将剧本文本进一步解析成分镜级结构化数据**
- **支持生成镜头图像、角色配音、旁白音频和镜头视频片段**
- **支持字幕、时间线和最终成片渲染**
- **支持通过 Flask 接口把 script-to-video 能力接入原有产品链路**

如果从工程角度总结，这个项目已经不是“只会产出文本”的 Demo，而是开始具备：

- **文本内容生产能力**
- **结构化中间表示能力**
- **多模态资产编排能力**
- **最终交付能力**

### 5. 可面试展示的效果表达
如果面试官问“你这个项目现在做到什么程度”，可以概括为：

> 我先把短剧剧本生成链路做成多 Agent 的工程化系统，然后进一步把剧本升级成视频友好的中间表示，已经打通从用户自定义输入，到剧本生成、镜头解析、TTS 配音、关键帧生成、镜头视频渲染，再到字幕和最终成片输出的 MVP 链路。这个过程里我重点处理了长文本可控性、分镜结构化、旁白策略以及音画同步问题。

## 应聘大模型应用开发工程师岗位面试可能遇到的问题
### 1. 为什么选择多 Agent，而不是单 Agent 或单 Prompt？
参考回答：  
这个项目之所以选择多 Agent，而不是单 Prompt，核心原因是短剧剧本生成属于长文本、强结构、强约束任务。单 Prompt 虽然可以快速生成一版内容，但很难同时兼顾大纲设计、正文生成、质量检查、问题修复和最终交付。多 Agent 的价值在于把复杂任务拆成几个更明确的子任务：Planner 负责结构化规划，Writer 负责正文生成，Reviewer 负责审校，Director 负责状态流转和修订策略。这样做不是为了“把系统做复杂”，而是为了提高长链路任务的稳定性、可控性和后续维护能力。

### 2. 你怎么判断一个大模型应用不是 Demo，而是真正可落地？
参考回答：  
我一般会从五个维度判断一个大模型应用是否真正可落地：第一，是否有失败处理；第二，是否有状态追踪；第三，是否有质量校验；第四，是否有可观测性；第五，是否具备真实交付链路。这个项目里，这五点都有对应实现。比如不完整剧本会被标记为失败，不会误报成功；系统有 JobStore 和运行日志追踪任务状态；有 Reviewer 和 `script_status` 来判断结构与完整性；前端能轮询展示整个生成过程；最终还能导出 PDF / Word，说明它不是只停留在“生成一段文本”的 Demo，而是具备业务交付能力的应用系统。

### 3. LangGraph 比自己写流程控制代码好在哪里？
参考回答：  
这个项目里我选择 LangGraph，主要是因为剧本生成流程不是一个简单的线性链路，而是一个有状态、有条件分支的工作流。比如审校通过就结束，审校不通过可能进入重写，某些情况只需要补尾，达到修订上限后又要输出当前最优稿。自己写流程控制当然也能实现，但随着分支变多，很快就会演变成难维护的手写状态机。LangGraph 的好处是能把状态、节点和边表达得很清楚，状态统一、条件路由清晰，后续扩展新的 Agent 也更自然，特别适合多节点、多分支、有修订回路的应用场景。

### 4. 你在这个项目里怎么做性能优化？
参考回答：  
这个项目里我主要从几个方向做性能优化。第一，Planner 阶段原本在 JSON 解析失败后会再触发一次远程修复请求，后来我改成单次生成加本地 JSON 解析，失败就快速 fallback，这样显著减少了规划阶段的等待时间。第二，Writer 阶段根据目标场景数动态控制 token，避免单集场景浪费过多输出预算，同时给多集长文本足够空间。第三，Reviewer 审校阶段增加了本地 fastpath，如果本地规则已经确认通过，就直接跳过远程润色。第四，Director 对修订次数做上限控制，避免系统陷入长时间循环重试。整体思路是减少无效模型调用、缩短关键链路，并让不同场景使用更合适的资源配置。

### 5. 你怎么处理长文本生成截断问题？
参考回答：  
长文本截断是这个项目里一个很核心的问题。我采用的是“规则识别 + 定向修复”的方案。先通过规则判断剧本是否可能截断，比如检查结尾是否完整、目标场景数和实际场景数是否一致、最后一场是否具备动作描述、对白和分镜提示等结构。如果发现只是尾部缺失，就优先走定向补尾，而不是整稿重写，这样成本更低，也不容易引入前文重复。另外，不完整结果不会被标记成成功，而是保留部分内容并返回失败态，避免系统把有问题的结果误当成可交付稿件。

### 6. 你怎么理解“大模型应用开发工程师”和“算法研究”的区别？
参考回答：  
我理解“大模型应用开发工程师”和“算法研究”最大的区别，在于关注点不同。算法研究更偏模型本身，比如预训练、微调、训练策略、评测体系；而大模型应用开发更关注模型怎么接入真实业务系统，怎么做工作流编排、质量治理、失败处理、交付链路、监控和成本优化。这个项目明显更偏应用开发方向，因为我做的重点不是训练模型，而是设计 Agent 工作流、封装模型调用服务、处理长文本稳定性、修复失败态、支持文档导出和前后端联调。从岗位匹配上看，它更能体现大模型应用工程能力。

### 7. 如果面试官问：这个项目最大的难点是什么？
参考回答：  
我认为这个项目最大的难点，不是“怎么让模型写出一段剧本”，而是“怎么让它稳定地产出完整、可交付的多集剧本”。真正有挑战的是长文本控制、尾部截断、修订成本、失败态识别，以及前后端对结果状态的一致理解。比如如果剧本只生成到一半，但前端却把它当成成功结果展示，这在业务上就是很严重的问题。所以这个项目的核心难点其实是工程化治理，而不只是 Prompt 编写。

### 8. 如果让你继续优化这个项目，你会做什么？
参考回答：  
如果继续优化这个项目，我会从几个方向推进。第一，引入检索增强，把题材规则、人物设定、行业模板接进来，提升生成的一致性和专业性。第二，把 Reviewer 进一步拆细，比如分成结构审校、风格审校、逻辑审校，让反馈更精确。第三，引入缓存和请求分级策略，进一步降低延迟和成本。第四，建立评测集和自动化回归测试，用更稳定的方式验证多集生成、补尾和失败态治理。第五，扩展更多交付形式，比如分镜卡、角色小传、拍摄脚本等，让系统更贴近真实内容生产流程。

### 9. 如果让你用一句话概括这个项目的技术亮点，你会怎么说？
参考回答：  
这是一个基于 LangGraph 编排多 Agent、面向长文本内容生产场景的大模型应用系统，我重点解决了剧本生成过程中的任务拆解、长文本完整性治理、失败态管理、性能优化以及文档交付落地问题。

## 视频化扩展方向：面试问答版
### 1. 为什么这个项目当前没有优先做视频化功能？
参考回答：  
当前项目没有优先做视频化，主要是因为这个阶段的核心目标是先把剧本生成链路做稳。对于短剧场景来说，文本生成本身就已经有很多工程问题需要解决，比如多集剧本的完整性、尾部截断、场景数控制、失败态识别、审校和补尾策略等。如果在这些问题还没有稳定之前就直接接视频化，问题会被进一步放大。文本里一个小问题，到了视频阶段就可能演变成镜头错乱、角色不一致、配音不同步、字幕错误或者整体节奏崩坏。所以从工程路径上，更合理的顺序是先把“内容生产”做稳，再把“多模态交付”接到后面。换句话说，视频化不是这个项目里一个简单的导出功能，而是下一阶段的独立系统建设。

### 2. 为什么视频化不能简单理解成“多一个导出按钮”？
参考回答：  
因为 PDF / Word 导出本质上是格式化交付，核心工作是把已经生成好的文本按规则排版成文档；但视频化不是格式转换，而是新增一整条多模态生成链路。它至少会涉及剧本结构化、分镜拆解、画面生成、角色一致性控制、配音、字幕时间轴、背景音乐、镜头合成和最终渲染。也就是说，视频化不是“把文本换个输出格式”，而是要把剧本变成一套可执行的镜头级数据，再驱动多个多模态模型和渲染模块完成最终产出。所以它的复杂度和普通导出完全不是一个量级。

### 3. 如果接入“剧本生成后一键生成视频”，你会怎么设计？
参考回答：  
如果让我扩展这个项目做一键生成视频，我不会直接把整份剧本丢给视频模型，而是会采用分层设计。第一层是剧本结构化，把剧本文本转成场景、镜头、人物、动作、对白、情绪、景别、运镜等结构化数据；第二层是分镜规划，新增 Storyboard Agent 或 Shot Planner Agent，把每个场景拆成更适合视频生成的镜头卡片；第三层是视觉生成，为每个镜头生成视觉提示词，可以走“关键帧/角色图 + 图生视频”或者“直接文生视频”两种路线；第四层是音频生成，包括 TTS 配音、情绪控制、背景音乐和环境音；第五层是时间轴构建，把镜头时长、台词、字幕、配音和转场映射到统一 timeline；第六层是视频合成，用程序化方式完成字幕叠加、音频混轨和最终渲染。整体上我会坚持一个原则：LLM 负责规划和中间表示生成，工程模块负责渲染和交付。

### 4. 如果扩展视频化能力，系统里应该新增哪些 Agent 或模块？
参考回答：  
如果扩展到视频化，我会在现有基础上新增几类角色。第一类是 Storyboard Agent，负责把剧本进一步拆成镜头级脚本；第二类是 Visual Prompt Agent，负责把镜头描述转成适合视觉模型的提示词，并尽量保持角色和风格一致；第三类是 Voice Agent，负责角色配音、语速、情绪和音频规划；第四类是 Video QC Agent，负责检查字幕、配音、镜头衔接和成片质量。除了 Agent，还需要一些工程模块，比如 Asset Manager 管理角色图和场景素材，Timeline Builder 生成时间轴，Video Render Service 完成视频合成，以及任务队列来支持长时异步渲染。也就是说，视频化更像是在现有多 Agent 文本系统后面，增加一条多模态生产流水线。

### 5. 你觉得视频化方案里最大的技术难点是什么？
参考回答：  
我认为最大难点不在“能不能生成一个视频片段”，而在“能不能稳定生成一组连续、统一、可交付的视频镜头”。这里最难的几个点包括：第一，角色一致性，前后镜头里人物的脸、服装、发型、气质容易漂移；第二，场景连续性，相邻镜头的背景、道具、光线和人物站位可能不一致；第三，时间轴同步，字幕、配音、镜头时长和转场必须严格匹配；第四，成本和时延，视频生成远比文本生成更贵、更慢；第五，失败恢复，如果某个镜头失败，怎么只重试局部而不破坏整体一致性。这些问题说明视频化的核心难点是多模态系统工程，而不只是模型调用。

### 6. 角色一致性为什么会成为视频化的核心问题？
参考回答：  
因为在剧本里，“同一个角色”只是一段文本描述，但在视频生成里，它必须变成前后一致的视觉实体。比如女主在第一镜是黑长发、蓝布裙、清冷狠厉，后面镜头如果变成另一张脸、另一套衣服，用户会直接觉得生成结果不可用。文本生成里角色一致性主要是语义问题，而视频生成里它会变成图像和视频的一致性问题。要解决这个问题，通常需要角色设定卡、参考图、固定视觉提示词，甚至引入 LoRA、reference image 或角色 embedding 之类的机制。所以角色一致性会是视频化落地里最先要解决的核心问题之一。

### 7. 和直接文本生成相比，视频化为什么更难做质量评估？
参考回答：  
文本质量还能通过一些规则和审校逻辑来判断，比如场景数是否达标、结尾是否完整、关键词是否覆盖、格式是否合规；但视频质量更复杂，很多问题不能只靠规则识别。比如角色像不像同一个人、镜头转场自然不自然、情绪表达是否到位、画面节奏是否合理、爽点是否打出来，这些都比文本质量评估更主观、更难自动化。所以如果项目扩展到视频化，我会考虑把评估拆成几层：先做规则层检查，比如字幕是否完整、音画是否同步、镜头数是否对齐；再结合模型评估和人工抽检去看整体观感。视频化真正难的不是生成，而是稳定地评估和控制成片质量。

### 8. 如果面试官问：你为什么现在不先把视频化做出来？
参考回答：  
我会回答说，这不是做不做的问题，而是产品阶段和工程优先级的问题。当前项目的核心目标是验证并稳定“短剧剧本生成”这个内容生产链路，而不是一次性把文本、音频、视频全打通。如果当前文本链路还不稳定，视频化只会放大已有问题，并显著增加成本、时延和失败点。从应用开发角度，先把剧本生成、审校、失败态治理、文档交付这些基础环节做扎实，是更合理的路径。等文本链路足够稳定之后，再接入视频化，成功率和工程效率都会更高。

### 9. 如果让你用一句话概括这个项目未来的视频化扩展方向，你会怎么说？
参考回答：  
未来如果扩展视频化，我会把它设计成“以剧本结构化为中间表示、以多 Agent 负责分镜规划与质量控制、以工程模块负责多模态渲染与交付”的一键短剧生成系统。
